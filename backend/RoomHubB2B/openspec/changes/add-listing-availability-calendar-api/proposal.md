## Why

Арендатору нужно видеть свободные часы помещения до отправки заявки: RoomHub рассчитан на аренду на несколько часов или дней, поэтому календарь только с дневными статусами скрывает полезную доступность внутри частично занятого дня. API почасовых интервалов даст frontend единый источник доступности выбранного объявления и заложит основу для последующего бронирования.

## What Changes

- Добавить публичный read-only endpoint `GET /api/listings/{listingId}/availability` с обязательными параметрами `from` и `to` в формате локальной даты и времени ISO 8601 `YYYY-MM-DDTHH:00`.
- Возвращать упорядоченный массив занятых полуоткрытых интервалов `[startAt, endAt)` с границами, выровненными на начало часа; время вне этих интервалов считается доступным.
- Проверять существование и опубликованный статус объявления, корректность и часовую точность границ, порядок дат-времени и допустимую длину диапазона.
- Считать `from`, `to`, `startAt` и `endAt` локальным временем помещения без UTC offset и без преобразования относительно часового пояса сервера. До появления `ZoneId` объявления хранить эти значения в PostgreSQL как `TIMESTAMP WITHOUT TIME ZONE`.
- Описать локальное почасовое время в OpenAPI как `type: string` с pattern для `YYYY-MM-DDTHH:00`, примером и явным описанием семантики без UTC offset, а не как стандартный `format: date-time`.
- Добавить OpenAPI-аннотации для controller, query-параметров, ответов `200`/`400`/`404` и DTO интервалов, чтобы runtime-контракт `/api/openapi` полностью описывал новый endpoint.
- Расширить `OpenApiContractTests` проверками path `/api/listings/{listingId}/availability`, параметров `listingId`/`from`/`to`, response schema и полей `busyIntervals`.
- После реализации выполнить `mvnw.cmd verify -Popenapi-export`, обновить `openapi/roomhub-b2b.openapi.json` и проверить, что diff содержит новый endpoint и связанные схемы.
- Не включать в изменение создание и редактирование блокировок, заявки и бронирования: это отдельные возможности после MVP календаря.

Пример запроса:

```http
GET /api/listings/42/availability?from=2026-07-01T09:00&to=2026-07-03T18:00
```

Пример ответа:

```json
{
  "listingId": 42,
  "from": "2026-07-01T09:00",
  "to": "2026-07-03T18:00",
  "busyIntervals": [
    {
      "startAt": "2026-07-01T12:00",
      "endAt": "2026-07-01T15:00"
    },
    {
      "startAt": "2026-07-02T10:00",
      "endAt": "2026-07-03T11:00"
    }
  ]
}
```

## Capabilities

### New Capabilities

- `listing-availability`: Чтение занятых почасовых интервалов опубликованного объявления за заданный диапазон даты и времени.

### Modified Capabilities

Нет.

## Impact

- API: новый добавочный endpoint с почасовым контрактом; контракт существующего `GET /api/listings` не меняется, поэтому frontend не должен получить регрессию.
- Backend: новый подпакет доступности объявления, DTO ответа, почасовая валидация, объединение интервалов, обработка отсутствующего объявления и OpenAPI-аннотации нового контракта.
- База данных: новая таблица `listing_unavailability_periods` с внешним ключом на `listings` и колонками `start_at`/`end_at` типа `TIMESTAMP WITHOUT TIME ZONE`; схема создаётся Flyway-миграцией.
- OpenAPI: runtime-документ `/api/openapi`, contract-тест и версионируемый файл `openapi/roomhub-b2b.openapi.json` расширяются новым path, параметрами, ответами и схемами почасовой доступности.
- Тестирование: `OpenApiContractTests` фиксирует присутствие и форму нового публичного контракта; Maven-профиль `openapi-export` подтверждает соответствие экспортируемого JSON runtime-документу.
- Зависимости: новые внешние библиотеки не требуются.
