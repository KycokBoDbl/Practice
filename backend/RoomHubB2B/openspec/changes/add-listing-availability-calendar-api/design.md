## Context

Backend сейчас содержит только чтение опубликованных объявлений через слои controller, service и Spring Data JPA repository. Моделей заявок, бронирований и календаря нет. Объявление уже содержит почасовую стоимость, а продукт поддерживает аренду на несколько часов или дней, поэтому доступность должна сохранять и возвращать интервалы внутри календарного дня.

Доступность зависит от объявления и хранится в PostgreSQL. Запрос публичный, но данные неопубликованных объявлений не должны раскрываться. Диапазон ограничен 93 днями, чтобы объём ответа и стоимость overlap-запроса оставались предсказуемыми.

В модели объявления пока нет часового пояса. До появления такого поля дата и время интерпретируются как локальное время помещения без UTC-смещения.

## Goals / Non-Goals

**Goals:**

- предоставить стабильный read-only REST-контракт почасовой доступности опубликованного объявления;
- хранить занятость компактными полуоткрытыми интервалами локальной даты и времени;
- гарантировать часовую точность границ хранения и API;
- получить пересекающиеся интервалы одним запросом, обрезать их по диапазону запроса и сформировать упорядоченный нормализованный ответ;
- покрыть контракт unit/integration-тестами и применить схему через Flyway.

**Non-Goals:**

- API арендодателя для создания, изменения и удаления периодов недоступности;
- интервалы с точностью меньше одного часа;
- конвертация между часовыми поясами;
- заявки, подтверждение аренды, защита от конкурентного бронирования и платежи;
- рекомендации альтернативных объявлений;
- аутентификация и авторизация.

## Decisions

### 1. Endpoint использует полуоткрытый диапазон локальных дат-времени

Контракт: `GET /api/listings/{listingId}/availability?from=YYYY-MM-DDTHH:00&to=YYYY-MM-DDTHH:00`. Ответ содержит границы запроса и упорядоченный массив `busyIntervals`. Время в диапазоне запроса, не входящее ни в один занятый интервал, считается доступным.

Границы имеют тип `LocalDateTime`, должны приходиться на начало часа и образуют полуоткрытый диапазон `[from, to)`. Значение `to` не входит в диапазон, поэтому интервалы `10:00-11:00` и `11:00-12:00` соприкасаются без пересечения. Это устраняет неоднозначность включительной конечной границы и подходит будущим почасовым бронированиям.

Альтернатива с одним элементом статуса на каждый час отвергнута: запрос на 93 дня создавал бы до 2232 элементов даже при полностью свободном помещении. Компактный список занятых интервалов меньше и позволяет frontend вычислять свободные часы как дополнение.

### 2. Недоступность хранится часовыми интервалами, свободное время не хранится

Flyway создаёт таблицу `listing_unavailability_periods`:

- `id BIGSERIAL PRIMARY KEY`;
- `listing_id BIGINT NOT NULL REFERENCES listings(id) ON DELETE CASCADE`;
- `start_at TIMESTAMP WITHOUT TIME ZONE NOT NULL`;
- `end_at TIMESTAMP WITHOUT TIME ZONE NOT NULL`;
- `created_at TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP`;
- ограничение `CHECK (end_at > start_at)`;
- ограничения `CHECK (start_at = date_trunc('hour', start_at))` и `CHECK (end_at = date_trunc('hour', end_at))`;
- индекс по `listing_id`, `start_at`, `end_at`.

Репозиторий выбирает только интервалы с пересечением по условию `start_at < :to AND end_at > :from`. Равенство границ не является пересечением. Свободное время вычисляется как отсутствие пересекающегося занятого интервала.

Альтернатива с отдельными колонками `date`, `start_hour` и `end_hour` отвергнута: она усложняет интервалы через полночь и многодневную аренду. Единые timestamp-границы одинаково представляют интервал внутри дня и интервал на несколько дней.

### 3. Ответ обрезает и объединяет хранимые интервалы

Сервис сортирует найденные интервалы по `startAt`, обрезает каждый до `[from, to)` и объединяет пересекающиеся или соседние интервалы. Например, сохранённые интервалы `10:00-12:00` и `12:00-14:00` возвращаются как `10:00-14:00`.

Нормализация делает контракт детерминированным, не раскрывает дублирующиеся строки хранения и уменьшает объём ответа. Запрет пересечений на уровне БД не вводится, потому что у MVP нет write API и источники будущей занятости ещё не определены.

### 4. Логика размещается в подпакете доступности объявления

Классы календаря размещаются в `ru.esie.practice.roomhubb2b.listing.availability`, сохраняя связь с aggregate объявления и не перегружая существующий `ListingService`. Планируемые компоненты: entity интервала, repository с overlap-запросом, service, controller и response DTO.

Сервис сначала проверяет объявление через `ListingRepository` с условием `id` и `PUBLISHED`, затем загружает пересекающиеся интервалы одним запросом и нормализует их в памяти. Это исключает N+1-запросы. Отдельный верхнеуровневый модуль отвергнут как избыточный для capability, принадлежащей объявлению.

### 5. Время хранится без часового пояса до появления зоны объявления

API и entity используют `LocalDateTime`, PostgreSQL использует `TIMESTAMP WITHOUT TIME ZONE`. Значения означают локальное время помещения и не конвертируются относительно часового пояса сервера.

`TIMESTAMP WITH TIME ZONE` отвергнут на текущем этапе: без сохранённого `ZoneId` объявления backend не может надёжно преобразовать локальные часы помещения в конкретный момент UTC, особенно при переходах летнего времени. Добавление часового пояса объявления должно быть отдельным изменением с миграцией существующих интервалов.

### 6. Ошибки отображаются на стандартные HTTP-статусы

Spring MVC преобразует отсутствующие или неразбираемые query-параметры в `400 Bad Request`. Сервис отдельно валидирует выравнивание на начало часа, строгий порядок границ и лимит 93 дней. Отсутствующее и неопубликованное объявление дают одинаковый `404 Not Found`, чтобы публичный API не раскрывал скрытые записи.

Для MVP допустимо использовать `ResponseStatusException`; если в проекте появится общий формат ошибок, endpoint должен быть переведён на единый `@RestControllerAdvice` отдельным изменением.

### 7. Тесты разделяются по ответственности

Unit-тест сервиса проверяет часовую точность, полуоткрытые границы, свободный диапазон, обрезку, сортировку, объединение и лимит 93 дня. MVC-тест проверяет формат ответа и статусы `400`/`404`. Repository integration-тест с PostgreSQL проверяет overlap-запрос и ограничения миграции.

## Risks / Trade-offs

- [Локальное время неоднозначно без часового пояса] → До появления `ZoneId` значения явно трактуются как часы помещения; API не принимает смещения и не обещает UTC-семантику.
- [Периоды могут перекрываться в таблице] → Сервис объединяет их перед выдачей; ограничение пересечений можно добавить вместе с write API и правилами конкурентного бронирования.
- [Точность только до часа не поддерживает аренду на 30 минут] → Ограничение соответствует текущему требованию; переход к меньшему шагу потребует изменения check-ограничений и валидации API.
- [Read-only API не даёт арендодателю управлять данными] → Таблица и тестовые данные создают основу; write API проектируется отдельно с авторизацией.
- [Большое число интервалов увеличит стоимость сортировки и слияния] → Запрос ограничен 93 днями и извлекает только пересечения по индексированным колонкам.

## Migration Plan

1. Применить добавочную Flyway-миграцию таблицы, check-ограничений и индекса.
2. Развернуть backend с новым endpoint; существующие API и таблица `listings` не меняются.
3. При необходимости добавить отдельной миграцией демонстрационные почасовые интервалы для существующих тестовых объявлений.
4. Для отката вернуть предыдущую версию приложения. Добавочная неиспользуемая таблица может остаться; её удаление выполняется только отдельной согласованной миграцией, чтобы не потерять данные.

## Open Questions

Нет блокирующих вопросов для read-only MVP. Часовой пояс объявления должен быть определён до реализации бронирований в нескольких часовых поясах.
