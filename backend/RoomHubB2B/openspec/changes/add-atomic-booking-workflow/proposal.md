## Why

RoomHub уже показывает опубликованные площадки и их занятые интервалы, но не позволяет провести аренду от заявки до завершения. Нужен управляемый процесс бронирования, который запрещает недопустимые переходы статуса и не допускает двойного резервирования одного временного интервала при конкурентных запросах.

## What Changes

- Добавить сделку бронирования с состояниями `REQUESTED`, `AWAITING_CONFIRMATION`, `CONFIRMED`, `IN_PROGRESS`, `COMPLETED`, `REJECTED`, `CANCELLED` и `EXPIRED`.
- Добавить команды создания и чтения бронирования, одобрения, отклонения, подтверждения и отмены. Произвольная запись поля `status` через CRUD API не предоставляется.
- При одобрении атомарно резервировать почасовой полуоткрытый интервал `[startAt, endAt)`. Пересекающиеся заявки могут существовать одновременно, но только первая успешно одобренная заявка получает слот; остальные получают конфликт.
- Автоматически переводить неподтверждённое одобренное бронирование в `EXPIRED` после срока подтверждения и освобождать его интервал.
- Автоматически переводить подтверждённую сделку в `IN_PROGRESS` при наступлении `startAt` и в `COMPLETED` при наступлении `endAt`.
- Фиксировать почасовую ставку и итоговую стоимость в момент создания заявки, чтобы последующее изменение объявления не меняло условия сделки.
- Записывать историю каждого успешного перехода состояния.
- Включать активные резервы сделок в существующую выдачу занятых интервалов площадки.

Пример создания заявки:

```http
POST /api/bookings
Content-Type: application/json

{
  "listingId": 42,
  "startAt": "2026-07-10T10:00",
  "endAt": "2026-07-10T13:00"
}
```

```json
{
  "id": 81,
  "listingId": 42,
  "status": "REQUESTED",
  "startAt": "2026-07-10T10:00",
  "endAt": "2026-07-10T13:00",
  "pricePerHour": 2500.00,
  "totalPrice": 7500.00,
  "confirmationDeadline": null
}
```

Пример одобрения с удержанием слота:

```http
POST /api/bookings/81/approve
```

```json
{
  "id": 81,
  "status": "AWAITING_CONFIRMATION",
  "confirmationDeadline": "2026-07-01T18:30:00"
}
```

## Capabilities

### New Capabilities

- `booking-workflow`: Создание сделок бронирования, разрешённые переходы конечного автомата, атомарное удержание и освобождение интервалов, автоматические временные переходы, расчёт стоимости и история статусов.

### Modified Capabilities

Отсутствуют. Основная директория `openspec/specs` пока не содержит синхронизированной capability календаря; интеграция сделок с текущим availability API описывается в новой capability.

## Impact

- API: новые endpoints `/api/bookings` и командные endpoints `/api/bookings/{bookingId}/...`; существующие JSON-контракты `/api/listings` и `/api/listings/{listingId}/availability` не меняются.
- База данных: новые таблицы бронирований и истории переходов; существующая таблица `listing_unavailability_periods` расширяется ссылкой на бронирование и типом блокировки либо используется как единый календарь резервов.
- Backend: новый доменный модуль booking, транзакционные команды переходов, плановая обработка просроченных и наступивших сделок, интеграция с listing availability.
- OpenAPI: экспортируемый контракт дополняется новыми операциями и схемами.
- Frontend: существующие экраны не ломаются; новые операции требуют отдельного интерфейса оформления и обработки заявок.
- Внешняя оплата, аутентификация, проверка полномочий арендатора/арендодателя и документооборот не входят в этот MVP.
