## Context

Текущий backend предоставляет опубликованные объявления и read-only календарь, построенный по таблице `listing_unavailability_periods`. Сущности сделки, команд изменения состояния и защиты записи от конкурентного бронирования нет. `ListingStatus` и ограничение из `V5__add_listing_enum_constraints.sql` относятся только к жизненному циклу объявления и не должны использоваться для статусов сделки.

Изменение затрагивает REST API, новый доменный модуль booking, PostgreSQL-схему, существующий календарь и фоновую обработку переходов по времени. В MVP отсутствуют аутентификация и реальные компании, поэтому endpoints считаются доверенными командами соответствующих участников процесса, но доменные правила переходов применяются полностью.

## Goals / Non-Goals

**Goals:**

- Реализовать явный конечный автомат сделки без универсального setter статуса.
- Атомарно выдавать один слот при конкурентном одобрении пересекающихся заявок.
- Автоматически освобождать неподтверждённое удержание и продвигать подтверждённую сделку по времени.
- Сохранить цену сделки и полную историю переходов.
- Показать активные удержания и подтверждённые сделки в существующем availability API без изменения его JSON-схемы.
- Сделать временную логику детерминированно тестируемой.

**Non-Goals:**

- Аутентификация, авторизация и модель юридических лиц.
- Платёжный шлюз, возвраты, договоры и закрывающие документы.
- Изменение жизненного цикла объявления или ограничения `chk_listings_status`.
- Редактирование интервала или цены существующей заявки.
- Распределённые очереди и несколько независимо масштабируемых сервисов.

## Decisions

### 1. Booking является отдельным агрегатом

Таблица `bookings` хранит `listing_id`, статус, `[start_at, end_at)`, снимок `price_per_hour`, `total_price`, `confirmation_deadline`, timestamps и поле версии. Статусы объявления и сделки представлены разными enum и разными CHECK-ограничениями.

Это не перегружает `listings.status`, который сейчас допускает только `PUBLISHED`, и позволяет независимо развивать модерацию площадок и сделки. Альтернатива с добавлением booking-состояний в `ListingStatus` отклонена как смешивающая два жизненных цикла.

### 2. Переходы выполняются командами

API предоставляет:

- `POST /api/bookings`;
- `GET /api/bookings/{bookingId}`;
- `GET /api/bookings/{bookingId}/history`;
- `POST /api/bookings/{bookingId}/approve`;
- `POST /api/bookings/{bookingId}/reject`;
- `POST /api/bookings/{bookingId}/confirm`;
- `POST /api/bookings/{bookingId}/cancel`.

Каждая команда вызывает доменный переход, проверяющий текущее состояние. Повтор уже успешно выполненной команды идемпотентен и не создаёт вторую запись истории; другая недопустимая команда возвращает `409 Conflict`. Универсальный `PATCH status` не создаётся.

Разрешённые переходы:

```text
REQUESTED --approve--> AWAITING_CONFIRMATION --confirm--> CONFIRMED
    |                         |                           |
    +--reject--> REJECTED     +--expire--> EXPIRED       +--start--> IN_PROGRESS
    |                         |                           |               |
    +--cancel--> CANCELLED    +--cancel--> CANCELLED     +--cancel       +--finish--> COMPLETED
                                                          before start
```

### 3. Резерв создаётся при одобрении

Созданная заявка `REQUESTED` не блокирует календарь, поэтому арендодатель может получить несколько конкурирующих заявок. В транзакции `approve` сервис:

1. блокирует строку `listings` через `SELECT ... FOR UPDATE`;
2. повторно проверяет `PUBLISHED`, будущее время и отсутствие пересекающегося `listing_unavailability_periods`;
3. вставляет календарный блок, связанный с booking;
4. переводит booking в `AWAITING_CONFIRMATION`, устанавливает deadline и пишет историю;
5. фиксирует всё одной транзакцией.

Все будущие операции записи календаря должны брать ту же блокировку площадки. Такой подход защищает пересечения между ручными блоками и booking-блоками в одной таблице. Альтернатива с проверкой пересечения без блокировки отклонена из-за race condition; PostgreSQL exclusion constraint отклонён для MVP, потому что текущая модель допускает нормализацию исторически пересекающихся ручных интервалов.

### 4. Существующая таблица остаётся единым календарём

`listing_unavailability_periods` расширяется nullable-ссылкой `booking_id` с уникальностью и типом источника `MANUAL`/`BOOKING`. Для `BOOKING` ссылка обязательна, для `MANUAL` отсутствует. Блок создаётся при одобрении и удаляется при `EXPIRED` либо допустимой отмене. Для `CONFIRMED`, `IN_PROGRESS` и `COMPLETED` запись сохраняется, чтобы прошедший интервал оставался занят в календаре.

Отдельная таблица booking slots не используется: проверка пересечений между двумя таблицами не могла бы быть защищена одним обычным ограничением базы данных.

### 5. Удержание действует 30 минут, но не позже начала аренды

`confirmationDeadline = min(approvedAt + 30 minutes, startAt)`. Одобрение запрещено, если на момент получения блокировки площадки `startAt <= now`. Подтверждение после deadline не допускается: сначала выполняется переход в `EXPIRED` и освобождается календарь.

### 6. Временные переходы выполняются планировщиком и при чтении/команде

Периодический job пакетно обрабатывает:

- `AWAITING_CONFIRMATION` с `confirmationDeadline <= now` в `EXPIRED`;
- `CONFIRMED` с `startAt <= now` в `IN_PROGRESS`;
- `IN_PROGRESS` с `endAt <= now` в `COMPLETED`.

Перед чтением одной сделки или выполнением команды сервис также синхронизирует просроченное состояние этой сделки. Это уменьшает окно устаревшего статуса при задержке job. Если job обнаружил подтверждённую сделку уже после `endAt`, он последовательно записывает переходы `IN_PROGRESS` и `COMPLETED`.

Spring bean `Clock` используется вместо прямых вызовов системного времени. Обработка выполняется короткими транзакциями с блокировкой строки booking и повторной проверкой текущего состояния, поэтому параллельные job/API не дублируют переходы.

### 7. Цена фиксируется при создании

Для целого числа часов `totalPrice = listing.pricePerHour * hours`. Оба денежных значения сохраняются с точностью `NUMERIC(10,2)` для ставки и достаточной точностью для итоговой суммы. Изменение цены объявления после создания заявки не пересчитывает существующую сделку.

### 8. История переходов является append-only

`booking_status_history` содержит booking, nullable `from_status`, `to_status`, причину/команду и `created_at`. Создание записывает переход `null -> REQUESTED`. История сортируется по `created_at`, затем по id и доступна отдельным read endpoint.

## Risks / Trade-offs

- [Все writers календаря обязаны использовать блокировку listing] -> Инкапсулировать создание и удаление периодов в одном сервисе календаря и покрыть конкурентным интеграционным тестом.
- [Планировщик может выполняться с задержкой] -> Синхронизировать состояние при чтении и командах; считать timestamps источником истины.
- [Локальное время площадки сейчас не имеет timezone] -> Сохранить текущую семантику `TIMESTAMP WITHOUT TIME ZONE`; timezone площадки вынести в отдельное изменение.
- [Нет авторизации ролей] -> Явно не обещать проверку участника в контракте MVP; не добавлять фиктивные security-механизмы.
- [Удаление booking-блока теряет календарный след отменённого удержания] -> Историю процесса сохранять в `booking_status_history`; календарь отражает только фактическую занятость.
- [Блокировка listing сериализует одобрения одной площадки] -> Для MVP это осознанный обмен небольшой пропускной способности на простой и доказуемый инвариант.

## Migration Plan

1. Flyway создаёт `bookings` и `booking_status_history` с FK, CHECK и индексами.
2. Следующая часть миграции расширяет `listing_unavailability_periods` полями источника и booking-ссылкой; существующие строки получают источник `MANUAL`.
3. Backend разворачивается с новыми endpoints и выключенным по умолчанию коротким scheduler delay до успешной инициализации контекста.
4. Rollback приложения возможен, пока новые endpoints не используются; миграция остаётся forward-only, а старый backend игнорирует добавленные nullable-колонки и новые таблицы.

## Open Questions

- Длительность удержания `30 минут` должна стать конфигурационным свойством, но её продуктовое значение можно изменить без смены модели.
- После появления аутентификации потребуется определить, кто именно может выполнять approve/reject и confirm/cancel; это отдельная capability.
